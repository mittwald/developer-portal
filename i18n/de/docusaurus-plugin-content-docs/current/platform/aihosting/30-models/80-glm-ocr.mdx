Der [GLM-OCR-Guide](../../examples/glm-ocr/) enthält lauffähige Beispiele für alle gängigen Anwendungsfälle, darunter eine vollständige Pipeline mit [Qwen3-Embedding-8B](/docs/v2/platform/aihosting/models/qwen3-embedding-8b), [Qwen3.5-122B-A10B-FP8](/docs/v2/platform/aihosting/models/qwen3-5-122b-a10b-fp8), n8n und Vektordatenbank-Optionen (pgvector, Qdrant, ChromaDB).

:::note[Datenverarbeitung]
**Es werden keine Dokumentdaten auf Datenträgern gespeichert.** Der Proxy wandelt Dokumente im Arbeitsspeicher in PNG-Bilder um und übergibt sie direkt an das Modell — zwischen Anfragen wird nichts gespeichert. Externe URLs in HTML-, CSS- oder SVG-Dateien werden geblockt, um SSRF zu verhindern. Die gesamte Verarbeitung findet auf mittwald-Servern statt.
:::

## Beschreibung

„GLM-OCR" ist ein auf optische Zeichenerkennung (OCR) spezialisiertes Modell von Z.ai (ZhipuAI), das für die präzise Textextraktion aus Dokumenten und Bildern entwickelt wurde. Das Modell basiert auf der GLM-V-Encoder-Decoder-Architektur – bestehend aus einem 0,4B CogViT-Visual-Encoder und einem 0,5B GLM-Sprachdecoder – und erzielt trotz nur 0,9B Parametern State-of-the-Art-Ergebnisse (94,62 Punkte auf OmniDocBench V1.5, Platz 1 weltweit). Ein auf unserer Plattform integrierter Dokumenten-Proxy wandelt PDF-, DOCX-, PPTX-, XLSX-, HTML-, SVG- und Rasterbild-Formate automatisch in PNG-Seiten um, bevor das Modell sie verarbeitet.

Geeignet für und unterstützt:

- Textextraktion aus PDF-, DOCX-, PPTX-, XLSX-, HTML- und vielen weiteren Dokumentenformaten
- Verarbeitung gescannter Dokumente, Rechnungen, Verträge, Formulare und Berichte
- Texterkennung in Tabellen und strukturierten Layouts
- Erkennung von Formeln und mathematischen Ausdrücken
- Key Information Extraction (KIE) — strukturierte JSON-Ausgabe aus Formularen, Belegen, Zertifikaten und Ausweisen
- RAG-Vorverarbeitung (Retrieval-Augmented Generation) — hochgenaues Dokumenten-Parsing für Wissensdatenbanken
- Mehrsprachige Dokumente — Chinesisch, Englisch, Deutsch, Französisch, Spanisch, Russisch, Japanisch, Koreanisch und weitere Sprachen

Folgende Einschränkungen gelten:

- Maximal **30 Seiten** pro Anfrage – bei Überschreitung liefert die API HTTP 413. Teile größere Dokumente in Batches von je 30 Seiten auf.
- Maximale Anfragegröße: **200 MB**
- Maximale Kontextlänge: 131.072 Token (ca. 4.000 Token pro Seite bei typischer Dokumentdichte)
- Kein Tool-Calling oder Function-Calling
- Keine Konversations-/Mehrturngespräche – jede Extraktion ist eine zustandslose Einzelanfrage

## Unterstützte Eingabeformate

Alle Inhalte werden als Base64-kodierte Data-URI im Feld `image_url` übermittelt. Der Proxy erkennt das Format automatisch und wandelt es in seitenweise PNG-Bilder um, bevor sie an das Modell weitergeleitet werden.

| Format | MIME-Typ für Data-URI | Hinweise |
| ------ | --------------------- | -------- |
| PDF | `application/pdf` | Bis zu 30 Seiten pro Anfrage |
| JPEG | `image/jpeg` | Wird nativ verarbeitet |
| PNG | `image/png` | Wird nativ verarbeitet |
| TIFF | `image/tiff` | Mehrere Frames → eine Seite pro Frame |
| GIF | `image/gif` | Animiert → eine Seite pro Frame |
| WebP | `image/webp` | Animiert → eine Seite pro Frame |
| BMP | `image/bmp` | |
| SVG | `image/svg+xml` | Rasterisierung via cairosvg |
| HTML | `text/html` | Rendering via WeasyPrint |
| DOCX | `application/vnd.openxmlformats-officedocument.wordprocessingml.document` | Konvertierung via mammoth + WeasyPrint |
| PPTX | `application/vnd.openxmlformats-officedocument.presentationml.presentation` | Eine Seite pro Folie |
| XLSX | `application/vnd.openxmlformats-officedocument.spreadsheetml.sheet` | Eine Seite pro Tabellenblatt, max. 2.000 Zeilen |
| XLS | `application/vnd.ms-excel` | Älteres Excel-Format |

## API-Nutzung

GLM-OCR wird über den Standard-**Chat-Completions-Endpunkt** (`/v1/chat/completions`) mit dem Modellnamen `GLM-OCR` aufgerufen. Der Dokumenten-Proxy fängt die Anfrage ab, konvertiert das Dokument in PNG-Seiten und leitet sie an das Modell weiter – das manuelle Aufteilen von Seiten ist nicht erforderlich.

### PDF-Dokument extrahieren

#### Python

```python
import base64
from openai import OpenAI

client = OpenAI(
    base_url="https://llm.aihosting.mittwald.de/v1",
    api_key="<dein-api-key>",
)

with open("dokument.pdf", "rb") as f:
    pdf_b64 = base64.b64encode(f.read()).decode()

response = client.chat.completions.create(
    model="GLM-OCR",
    messages=[
        {
            "role": "user",
            "content": [
                {
                    "type": "image_url",
                    "image_url": {
                        "url": f"data:application/pdf;base64,{pdf_b64}",
                    },
                },
                {
                    "type": "text",
                    "text": "Extrahiere den gesamten Text aus diesem Dokument.",
                },
            ],
        }
    ],
    temperature=0.1,
)

print(response.choices[0].message.content)
```

#### JavaScript

```ts

import OpenAI from "openai";

      ],
    },
  ],
  temperature: 0.1,

```

### Office-Dokumente (DOCX, PPTX, XLSX)

Office-Dokumente verwenden dieselbe Anfragenstruktur – nur der MIME-Typ in der Data-URI ändert sich:

```python
import base64
from openai import OpenAI

client = OpenAI(
    base_url="https://llm.aihosting.mittwald.de/v1",
    api_key="<dein-api-key>",
)

MIME_TYPES = {
    ".docx": "application/vnd.openxmlformats-officedocument.wordprocessingml.document",
    ".pptx": "application/vnd.openxmlformats-officedocument.presentationml.presentation",
    ".xlsx": "application/vnd.openxmlformats-officedocument.spreadsheetml.sheet",
    ".html": "text/html",
    ".svg":  "image/svg+xml",
}

def dokument_extrahieren(pfad: str, anweisung: str = "Extrahiere den gesamten Text aus diesem Dokument.") -> str:
    ext = pfad.rsplit(".", 1)[-1].lower()
    mime = MIME_TYPES.get(f".{ext}", "application/pdf")
    with open(pfad, "rb") as f:
        b64 = base64.b64encode(f.read()).decode()
    response = client.chat.completions.create(
        model="GLM-OCR",
        messages=[{
            "role": "user",
            "content": [
                {"type": "image_url", "image_url": {"url": f"data:{mime};base64,{b64}"}},
                {"type": "text", "text": anweisung},
            ],
        }],
        temperature=0.1,
    )
    return response.choices[0].message.content

# Beispiele
print(dokument_extrahieren("vertrag.docx"))
print(dokument_extrahieren("praesentation.pptx"))
print(dokument_extrahieren("bericht.xlsx"))
```

## Empfohlene Inferenzparameter

GLM-OCR ist ein deterministisches Extraktionsmodell. Verwende niedrige Temperaturen für eine genaue, quellengetreue Textextraktion:

| Parameter     | Wert |
| ------------- | ---- |
| `temperature` | 0.1  |
| `top_p`       | 1.0  |
| `max_tokens`  | 4096 pro Seite (entsprechend der Seitenanzahl skalieren) |

## Ausgabemodi

Das Ausgabeformat des Modells wird ausschließlich über den Prompt gesteuert – es gibt keinen separaten API-Parameter dafür.

| Modus | Aktivierung | Verhalten |
| ----- | ----------- | --------- |
| **Klartext** | `"Extrahiere den gesamten Text aus diesem Dokument."` | Roher Text ohne Formatierung |
| **Markdown** | `"Extrahiere den Text und formatiere ihn als Markdown. Verwende # für Überschriften und - für Listen."` | Erhält Überschriften, Listen, Hervorhebungen – gut für RAG |
| **JSON (KIE)** | `"Extrahiere diese Felder und gib sie als JSON zurück: {…}"` | Strukturierte Extraktion; Ausgabe immer in ` ```json ``` `-Blöcken – vor dem Parsen entfernen |
| **HTML-Tabelle** | `"Gib die Tabelle als HTML-<table>-Element zurück."` | Nützlich für tabellenartige Daten |

:::note[Beobachtetes Modellverhalten]
- **JSON ist immer mit Code-Blöcken umschlossen.** Das Modell umschließt JSON-Antworten unabhängig vom Prompt mit Markdown-Code-Blöcken. Vor dem Aufruf von `json.loads()` mit `re.sub(r"^```[a-z]*\n?", "", raw.strip())` entfernen.
- **Markdown-Modus benötigt konkrete Anweisungen.** Ein allgemeines „Wandle in Markdown um" erzeugt keine zuverlässige Überschriften-Syntax – immer `# für Hauptüberschriften, ## für Unterüberschriften, - für Listen` angeben.
- **HTML-Eingabe → HTML-Tabellenausgabe.** Enthält die Eingabe eine HTML-Datei mit einer `<table>`, gibt das Modell diese als HTML aus – auch wenn Markdown-Tabellen angefordert werden. Für saubere Markdown-Tabellenausgabe eine PDF- oder Bildquelle verwenden.
- **XLSX-Ausgabe enthält den Tabellenblatt-Namen.** Bei Excel-Dateien gibt das Modell den Tabellenblatt-Namen als erste Zeile aus. Diese Zeile beim programmatischen Weiterverarbeiten herausfiltern.
- **PPTX und XLSX benötigen gültiges Open XML.** Beschädigte oder programmatisch erzeugte ungültige Office-ZIP-Strukturen führen zu HTTP 400.
:::

## Tipps für spezifische Aufgaben

### Große Dokumente (> 30 Seiten)

Teile das PDF in Batches auf und füge die Ergebnisse zusammen:

```python
import base64
from pypdf import PdfReader, PdfWriter
from io import BytesIO
from openai import OpenAI

client = OpenAI(
    base_url="https://llm.aihosting.mittwald.de/v1",
    api_key="<dein-api-key>",
)

def pdf_batch_extrahieren(pdf_bytes: bytes) -> str:
    pdf_b64 = base64.b64encode(pdf_bytes).decode()
    response = client.chat.completions.create(
        model="GLM-OCR",
        messages=[{
            "role": "user",
            "content": [
                {"type": "image_url", "image_url": {"url": f"data:application/pdf;base64,{pdf_b64}"}},
                {"type": "text", "text": "Extrahiere den gesamten Text aus diesem Dokument."},
            ],
        }],
        temperature=0.1,
    )
    return response.choices[0].message.content

def grosses_pdf_extrahieren(pfad: str, batch_groesse: int = 30) -> str:
    reader = PdfReader(pfad)
    ergebnisse = []
    for start in range(0, len(reader.pages), batch_groesse):
        writer = PdfWriter()
        for seite in reader.pages[start:start + batch_groesse]:
            writer.add_page(seite)
        buf = BytesIO()
        writer.write(buf)
        ergebnisse.append(pdf_batch_extrahieren(buf.getvalue()))
    return "\n\n".join(ergebnisse)
```

### Tabellen und strukturierte Daten

Für Dokumente mit Tabellen weise das Modell an, die Struktur beizubehalten:

```
Extrahiere den gesamten Text aus diesem Dokument. Erhalte Tabellenstrukturen mittels einfacher Textausrichtung oder Markdown-Tabellen, wo sinnvoll.
```

### Key Information Extraction (KIE)

Um strukturierte Felder aus Rechnungen, Formularen oder Belegen als JSON zu extrahieren, beschreibe das gewünschte Schema im Prompt. Das Modell gibt JSON in Markdown-Code-Blöcken zurück – diese müssen vor dem Parsen entfernt werden:

```python
import json
import re

response = client.chat.completions.create(
    model="GLM-OCR",
    messages=[{
        "role": "user",
        "content": [
            {"type": "image_url", "image_url": {"url": f"data:application/pdf;base64,{pdf_b64}"}},
            {
                "type": "text",
                "text": (
                    "Extrahiere die folgenden Felder aus dieser Rechnung und gib sie als JSON-Objekt zurück:\n"
                    '{ "rechnungsnummer": "", "datum": "", "lieferant": "", "gesamtbetrag": "", "positionen": [] }'
                ),
            },
        ],
    }],
    temperature=0.1,
)

raw = response.choices[0].message.content
# Markdown-Code-Blöcke entfernen, die das Modell um das JSON setzt
clean = re.sub(r"^```[a-z]*\n?", "", raw.strip())
clean = re.sub(r"\n?```$", "", clean)
data = json.loads(clean)
```

### Formel- und Ausdruckserkennung

Für Dokumente mit mathematischen Formeln kann LaTeX-Ausgabe angefordert werden:

```
Extrahiere den gesamten Text aus diesem Dokument. Gib mathematische Formeln und Ausdrücke in LaTeX-Notation aus: $...$ für Inline-Mathematik und $$...$$ für abgesetzte Formeln.
```

### Genauigkeit bei schlechter Scanqualität verbessern

Bei niedrig aufgelösten oder beschädigten Dokumenten kannst du das Modell explizit anweisen:

```
Extrahiere allen lesbaren Text aus diesem Dokument. Ist ein Wort unklar, übertrage deine beste Einschätzung und markiere sie mit [?].
```
